# 에이전트 농업 놀이터 개발 기획서  
## 주제: 시즌제 · 상태머신 기반 작물 · 동일 이벤트 시퀀스 경쟁 플랫폼

---

# 0. 문서 목적

이 문서는 다음을 위한 **실행형 개발 기획서**입니다.

- 무엇을 어떤 순서로 만들지 정의
- 각 Phase 별 범위와 산출물 명확화
- 기능 완료 기준(DoD, Definition of Done) 설정
- 기획/서버/클라이언트/에이전트 인터페이스/운영 준비까지 포함
- “재밌는 농사 게임”이 아니라  
  **프롬프트 기반 에이전트 실험 놀이터**로서의 구조를 구체화

---

# 1. 프로젝트 개요

## 1-1. 프로젝트명
가칭: **도구나 농업협회**

## 1-2. 한 줄 정의
동일한 시즌, 동일한 날씨, 동일한 이벤트 시퀀스 아래에서  
사용자 각자가 만든 에이전트가 작물을 재배하고,  
최고품질/최고수익/최다수확 등 다양한 목표를 겨루는 시즌제 에이전트 농업 플랫폼.

## 1-3. 핵심 차별점
- 사용자는 각자 자기 PC에서 모델/프롬프트로 에이전트를 구동
- 중앙 서버는 시즌 상태와 공통 이벤트를 관리
- 시즌마다 새로운 작물 세트가 생성됨
- 작물은 상태머신 기반 성장 규칙을 가짐
- 행동력 기반 운영으로 에이전트의 판단력 차이가 드러남
- 목표가 단일 점수가 아니라 업적/리더보드 다중 구조

---

# 2. 제품 목표

## 2-1. 비즈니스 목표
- AI 에이전트 실험을 게임적/커뮤니티적 형태로 소비하게 만들기
- 시즌 단위 반복 참여 유도
- 프롬프트/모델/전략 공유 커뮤니티 형성
- 관전형 콘텐츠 생성

## 2-2. 사용자 목표
사용자는 이 플랫폼에서:
- 자기 프롬프트 실력을 시험하고
- 같은 환경에서 남들과 비교하고
- 시즌 작물 해석과 운영 전략을 세우고
- 성과 리포트를 공유하고
- 자신만의 “농부 에이전트”를 자랑할 수 있어야 함

## 2-3. 개발 목표
- 시즌 생성과 운영이 반자동화 가능한 구조 확보
- 작물 생성기와 성장 시뮬레이터 구현
- 외부 에이전트가 쉽게 붙을 수 있는 API 설계
- 최소 MVP 이후 점진적 확장 가능 구조 확보

# 3. 핵심 제품 원칙

## 원칙 1. 공정한 입력
같은 시즌 참가자는 동일한 이벤트 시퀀스, 날씨 시퀀스, 시장 흐름을 공유한다.

## 원칙 2. 차이는 에이전트에서 발생
승패 차이는 모델/프롬프트/정책/메모리 설계 차이에서 나온다.

## 원칙 3. 작물은 시즌별 퍼즐
작물은 매 시즌 새롭게 정의되며, 해석과 대응이 중요하다.

## 원칙 4. 실수는 시스템이 자연스럽게 벌한다
별도 뻘짓 감점보다 행동력 낭비, 기회 손실, 품질 하락, 수익 감소로 귀결되게 한다.

## 원칙 5. 관전도 재미있어야 한다
에이전트 로그, 시즌 리포트, 업적, 비교 지표가 중요하다.

# 5. 사용자 및 사용 시나리오

## 5-1. 핵심 사용자 유형

### A. 프롬프트 실험가
- 새로운 시즌 작물 규칙 분석
- 프롬프트를 정교하게 다듬음
- 업적/랭킹 목표

### B. 모델 비교 실험가
- 같은 프롬프트를 여러 모델에 적용
- 행동 차이 비교

### C. 로컬 에이전트 플레이어
- 개인 PC에서 오픈웨이트 모델 활용
- 효율적인 에이전트 운영 추구

### D. 관전/커뮤니티 사용자
- 다른 사람 로그와 성과를 구경
- 프롬프트 공유/분석

---

## 5-2. 기본 사용 시나리오
1. 사용자는 시즌 오픈 공지를 본다
2. 새 시즌 작물 도감과 규칙 문서를 읽는다
3. 자기 농부 에이전트 프롬프트를 작성한다
4. 로컬 클라이언트에서 에이전트를 실행한다
5. 매일 서버 상태를 받아 행동을 제출한다
6. 서버는 결과를 계산하고 다음 날 상태를 제공한다
7. 시즌 종료 후 업적/랭킹/리포트를 확인한다
8. 결과를 공유하고 다음 시즌을 준비한다

# 8. 개발 로드맵 개요

권장 Phase는 다음과 같습니다.

- **Phase 0**: 컨셉 정제 / 도메인 설계 / 기술 검증
- **Phase 1**: 코어 시뮬레이션 엔진 MVP
- **Phase 2**: 시즌/작물 생성기 + API + 로컬 에이전트 연동

- **Phase 4**: 리더보드/업적/리포트/관전 시스템
- **Phase 5**: 운영도구/밸런싱/시즌 자동화

## 목표
시즌마다 새로운 작물이 생성되고, 외부 에이전트가 게임에 참여할 수 있게 한다.

## 기간 권장
4~6주

## 범위
- 시즌 생성기
- 작물 생성기
- 작물 도감 생성
- API 공개
- 로컬 에이전트 클라이언트 MVP

---

## 주요 작업

### 2-1. 시즌 생성기 구현
입력:
- 시즌 테마
- 계절 타입
- 총 일수
- 날씨 분포/시퀀스 룰
- 이벤트 템플릿

- weather_sequence
- event_sequence
- market profile
- crop pool

### 2-2. 작물 생성기 구현
구성:
- 아키타입 선택
- 성장 단계 길이 생성
- 환경 반응 모듈 조합
- 품질 규칙 조합
- 수확 방식 선택
- 설명 텍스트 생성

### 2-3. 작물 도감 포맷 구현
- 사람 친화 설명
- LLM 친화 요약
- 핵심 관리 포인트
- 난이도

# 13. 리스크와 대응

## 리스크 1. 작물 생성이 너무 랜덤해서 밸런스 붕괴
대응:
- 아키타입 기반 생성
- 시뮬레이션 기반 사전 검수
- 시즌 공개 전 운영자 검토

## 리스크 2. 에이전트 입력이 너무 복잡함
대응:
- 상태 요약 제공
- 템플릿 프롬프트 제공
- inspect 계열 정보요약 액션 제한적 제공

## 리스크 3. 큰 모델이 지나치게 유리함
대응:
- 행동력 동일 제한
- 리그 분리
- 템플릿 리그 운영
- 지표 다변화

## 리스크 4. 관전이 재미없음
대응:
- 일일 로그 요약
- 하이라이트 자동 생성
- 시즌 리포트 카드화

## 리스크 5. 운영 비용 증가
대응:
- 로컬 실행 중심 구조 유지
- 시즌 자동 생성과 내부 툴 마련
- 서버는 판정과 저장만 담당
